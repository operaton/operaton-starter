# Decision Log

## Session: 2026-06-08

### Context
New PRD created to capture UI enhancement requirements for operaton-starter.

---

### DEC-001: "Configure Now" flow — why Process Application is the default
"Configure Now" is the highest-visibility CTA and will be hit first by new users before they explore other options. Restricting it to a single project type defeats the purpose; it must be editable. "Process Application" is the sensible default because (a) it is the first project type shown and (b) it represents the most common new-user job.

### DEC-003: Footer version — sourced at build time
Version string is injected at build time (e.g. from Maven manifest/build property). The requirement is to strip `-SNAPSHOT` from the displayed value, not to maintain a static string manually.

### DEC-004: Tag chip color palette — UX implementation detail
Specific colors/hues per category are left to UX. Sally (UX) decides whether a legend is displayed. PRD specifies the category mapping only.

### DEC-005: groupId/artifactId defaults — freely overridable
`org.operaton.example` groupId and clean artifactId (no "example") are defaults only. Users can freely override them. The requirement is about the default values, not enforcement/validation.

### DEC-006: OQ-1 resolved — Tag type: direct switch, no transition
Project is not in production. The tag data model switches directly from `string[]` to `{ label: string, category: TagCategory }`. No backward-compatibility shim required.

### DEC-007: OQ-2 resolved — Version via Vite env var
Footer version is injected at build time via `import.meta.env.VITE_APP_VERSION`, sourced from the Maven POM via Vite/Maven resource filtering. Not exposed through the Metadata API.

### DEC-008: Version field validation — Maven pattern
The project version form field validates against Maven version format (non-empty, no whitespace).

### DEC-009: F2 bookmark/direct-URL behavior — explicit and correct
A user navigating directly to `/configure` (no `projectType` query param, e.g. from a bookmark) lands in the editable project type mode. This is the intended behavior, not an edge case.

### DEC-010: Tag unknown category fallback — grey chip
Tags with a missing or unrecognized category render as a grey/neutral chip.

### DEC-011: Footer missing-version fallback
If `VITE_APP_VERSION` is absent at build time, the footer shows "operaton-starter" without a version string.

### DEC-002: Scope of "Configure Now" — only project type is editable
Only the project type field becomes editable on the preview page when arriving via "Configure Now". All other fields behave as in the main-page → Configure flow. (To be confirmed: are other fields also unlocked, or strictly just project type?)
